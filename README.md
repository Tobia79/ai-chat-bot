# AI Chat Web 应用实战题

> 适用岗位：AI Agent / Agent Harness 研发实习生
> 建议形式：限时作业
> 建议投入：75 分钟
> 
> 

## 一、题目背景

实现一个可实际运行的 AI Chat Web 应用。

这道题不以页面复杂度为目标，重点考察候选人如何把模型能力组织成一个交互完整、状态可靠、便于维护和验证的产品。

技术栈可以自行选择，建议使用 React、Next\.js 或 Vue 构建前端，使用 Node\.js 或 Python 构建服务端。

## 二、核心要求

### 1\. 基础对话

- 用户可以发送消息并获得模型回复。

- 支持 Markdown、代码块和多行文本展示。

- 输入框支持 `Enter` 发送、`Shift + Enter` 换行。

### 2\. 流式输出

- 模型回复需要逐步展示，不能等待完整结果后一次性返回。

- 明确展示等待、生成中、完成、失败和已取消状态。

- 支持停止当前生成。

### 3\. 会话管理

- 支持创建、切换和删除会话。

- 切换会话后能够恢复历史消息。

- 浏览器刷新后，会话至少可以通过本地存储恢复。

### 4\. 异常与并发处理

- 网络异常或模型调用失败时，页面不能卡死。

- 提供可理解的错误提示和重试入口。

- 避免重复提交、消息乱序以及切换会话后的串流污染。

### 5\. 模型接入

- 通过服务端调用 OpenAI\-compatible API。

- API Key 只能通过环境变量配置，不得写入代码或暴露到浏览器。

- 提供 `.env.example`。

- 建议提供 Mock 模式，方便在没有真实 API Key 时运行和验收。

### 6\. 工程质量

- 前后端职责清晰，组件和状态组织合理。

- 至少为消息状态、流式解析或会话管理中的一个核心模块编写测试。

- 提供完整 README，包括启动方式、架构说明和已知限制。

## 三、最低验收场景

1. 创建会话并发送问题，可以看到流式回复。

2. 生成过程中点击停止，输出立即终止并进入可继续交互状态。

3. 切换到其他会话再返回，历史记录保持正确。

4. 刷新页面后，已有会话仍然存在。

5. 模型接口返回错误时，可以看到错误并重新请求。

6. 前端构建产物中不存在 API Key。

## 四、加分项

- 支持 Regenerate、编辑后重新发送或从指定消息分叉对话。

- 展示首 Token 延迟、总耗时、Token 用量、模型名称和 Request ID。

- 使用状态机或事件模型管理对话生命周期。

- 支持多个模型 Provider，并保持统一接口。

- 提供 Playwright E2E 测试。

- 实现一个简单 Tool Call，并展示调用过程与结果。

- 完善移动端布局和无障碍交互。

加分项不是完成作业的前置条件，不建议为了堆叠功能牺牲核心流程的正确性。

## 五、提交内容

- 可运行的代码仓库。

- README、`.env` 和必要的启动脚本。

- 一份简短的 `AI_WORKLOG.md`，说明使用了哪些 AI 编程工具、AI 帮助完成了什么，以及发现并修正过什么错误。

- 不需要提交完整 AI 对话记录，也不得提交任何密钥。
