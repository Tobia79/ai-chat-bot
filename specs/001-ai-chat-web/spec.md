## Clarifications

### Session 2026-07-15

- Q: 切换会话时，进行中的生成如何处理？ → A: 后台继续：流只写入原会话；切回可见进度/结果
- Q: 会话 A 后台生成时，能否在会话 B 再启动新生成？ → A: 全局同时仅一路；其他会话发送时阻止并提示。多会话并行生成（原选项 A）列为后续升级版本
- Q: 失败后的「重试」具体改写哪条消息？ → A: 就地重试：保留用户消息，替换/重跑失败的那条助手消息
- Q: 用户点击「停止」后，已流出的半段内容怎么处理？ → A: 保留已生成片段，消息状态为已取消
- Q: Mock 模式与真实密钥同时存在时以谁为准？ → A: 显式 Mock 优先；否则有密钥走真实，无密钥则报错提示

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 发送消息并获得流式回复 (Priority: P1)

用户打开应用后，在当前会话中输入问题并发送，能看到助手回复逐字（或逐段）出现；生成过程中可随时停止，停止后界面恢复可继续提问的状态。

**Why this priority**: 这是产品的核心价值；没有可靠的「问—答—停」闭环，其余功能无法验收。

**Independent Test**: 仅用单个默认会话，发送一条问题，观察流式展示、状态变化与停止后可继续发送即可独立验证。

**Acceptance Scenarios**:

1. **Given** 应用已就绪且模型服务可用（或 Mock 可用），**When** 用户创建或进入会话并发送一条问题，**Then** 界面先进入等待/生成中状态，随后内容逐步追加展示，最终进入完成状态。
2. **Given** 助手正在生成回复且已展示部分内容，**When** 用户点击停止，**Then** 输出立即停止，已生成片段保留在界面上，该助手消息状态变为已取消，输入区可再次发送新消息。
3. **Given** 用户准备发送多行内容，**When** 按下 `Shift + Enter`，**Then** 输入框换行且不会发送；**When** 按下 `Enter`（无 Shift），**Then** 消息被发送。
4. **Given** 助手回复包含 Markdown、代码块或多行文本，**When** 回复完成展示，**Then** 用户能正确阅读格式化内容与代码块。

---

### User Story 2 - 管理多个会话并恢复历史 (Priority: P2)

用户可创建多个独立会话、在会话间切换、删除不需要的会话；切换后历史消息保持正确；刷新浏览器后，已有会话与消息仍可恢复。

**Why this priority**: 会话管理直接对应最低验收场景中的切换与刷新恢复，是「可继续使用」的产品底线。

**Independent Test**: 创建两个会话并分别发消息，来回切换核对历史，再刷新页面核对持久化即可独立验证（可不依赖复杂流式异常场景）。

**Acceptance Scenarios**:

1. **Given** 用户已在会话 A 中有若干消息，**When** 用户新建会话 B 并发送消息，再切回会话 A，**Then** 会话 A 的历史完整且未被 B 污染。
2. **Given** 存在多个会话，**When** 用户删除其中一个会话，**Then** 该会话从列表消失且无法再切换到其历史；其他会话不受影响。
3. **Given** 浏览器中已有已保存的会话与消息，**When** 用户刷新页面，**Then** 会话列表与各会话历史仍可恢复并继续使用。

---

### User Story 3 - 失败可感知并可重试 (Priority: P2)

当网络异常或模型调用失败时，页面不卡死；用户能看到可理解的错误说明，并能对失败请求发起重试。

**Why this priority**: 作业明确要求异常不卡死与重试入口；直接影响可靠性评分与可验收性。

**Independent Test**: 在模型不可用或强制失败的条件下发送消息，确认错误提示与重试，且界面仍可操作。

**Acceptance Scenarios**:

1. **Given** 模型服务返回错误或网络不可用，**When** 用户发送消息，**Then** 界面展示失败状态与可读错误提示，且页面仍可继续操作。
2. **Given** 上一条助手消息已失败，**When** 用户触发重试，**Then** 系统保留对应的用户消息，在同一条助手消息上重新进入等待/生成流程（不追加新的用户消息副本）；最终完成、再次失败或被取消。

---

### User Story 4 - 并发与切换时状态不出错 (Priority: P1)

用户在生成过程中不会因重复点击导致重复提交；切换会话时，旧会话的流式内容不会写入当前会话；消息顺序保持正确。

**Why this priority**: 「串流污染 / 乱序 / 重复提交」是作业重点考察项，与流式正确性同分高权重。

**Independent Test**: 生成中快速连点发送、生成中切换会话再返回，核对消息归属与顺序。

**Acceptance Scenarios**:

1. **Given** 当前会话正在生成，**When** 用户再次点击发送，**Then** 系统不发起重复的有效生成请求（或明确忽略重复提交），界面不出现两条交叉错乱的回复。
2. **Given** 会话 A 正在流式生成，**When** 用户切换到会话 B，**Then** 会话 B 不被 A 的增量内容污染；A 的生成在后台继续仅写入 A；切回 A 后可见当时进度或已完成结果。
3. **Given** 会话 A 仍在生成中，**When** 用户在会话 B 尝试发送新消息，**Then** 系统不启动第二路生成，并向用户给出明确提示（例如说明需等待当前生成结束或先停止）。
4. **Given** 同一会话内连续多轮对话，**When** 各轮结束，**Then** 消息时间顺序与问答对应关系正确。

---

### User Story 5 - 密钥不落前端且可无密钥验收 (Priority: P2)

模型密钥仅由服务端通过环境配置读取，永不进入浏览器或前端构建产物；在没有真实密钥时，可通过 Mock 模式完成核心验收流程。

**Why this priority**: 安全性与可运行验收是硬性提交要求；Mock 降低验收门槛。

**Independent Test**: 检查环境示例配置与运行说明；无真实密钥时启用 Mock 走通 P1 主流程；核对前端产物中无密钥明文。

**Acceptance Scenarios**:

1. **Given** 部署/本地运行使用环境变量配置密钥且未启用 Mock，**When** 检查前端可见资源与构建产物，**Then** 不存在模型 API Key 明文。
2. **Given** 显式启用 Mock 模式（无论是否配置了真实密钥），**When** 用户发送消息，**Then** 使用 Mock 产生流式（或模拟流式）回复，并可完成停止、会话等核心交互验收。
3. **Given** 未启用 Mock 且未配置真实密钥，**When** 用户尝试发起需要模型的请求（或服务启动校验），**Then** 系统给出可理解的配置错误提示，且不静默失败或卡死。

---

### Edge Cases

- 用户在网络中断后恢复：失败提示可理解，就地重试或新发送可继续。
- 对失败助手消息就地重试：清空该助手消息的失败内容/错误态后重新生成，会话中不出现重复的用户气泡。
- 生成刚开始即点击停止：进入已取消；若尚无任何片段则可为空内容的已取消消息，无半残卡死状态。
- 停止时已有部分输出：保留片段并标为已取消，刷新后若已持久化则仍可见该片段与已取消状态。
- 删除当前正在查看的会话：自动落到其他会话或空状态，界面可用。
- 空消息或仅空白字符：不得发送有效请求。
- 超长回复：界面保持可滚动阅读，不导致页面无响应。
- 刷新发生在生成进行中：刷新后以本地已持久化内容为准；进行中未落盘的增量可丢失，但应用必须可继续使用（不得卡死）。
- 切开会话后后台生成仍在进行：仅更新原会话；当前可见会话不受影响。
- 全局已有一路生成进行中时，其他会话的发送被阻止并提示；用户可切回原会话查看进度，或先停止后再在其他会话发送。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 允许用户发送文本消息并获得助手回复。
- **FR-002**: 系统 MUST 以流式方式逐步展示助手回复，不得仅在全部生成结束后一次性呈现。
- **FR-003**: 系统 MUST 向用户明确展示对话生成生命周期中的状态，至少包括：等待、生成中、完成、失败、已取消。
- **FR-004**: 用户 MUST 能够停止当前正在进行的生成；停止后 MUST 保留已生成的文本片段，将该助手消息标记为已取消，并可继续发起新消息。
- **FR-005**: 输入区 MUST 支持 `Enter` 发送、`Shift + Enter` 换行。
- **FR-006**: 助手消息 MUST 支持 Markdown、代码块与多行文本的可读展示。
- **FR-007**: 用户 MUST 能够创建、切换与删除会话。
- **FR-008**: 切换会话后，系统 MUST 恢复该会话的历史消息且不得与其它会话混淆。
- **FR-009**: 浏览器刷新后，系统 MUST 至少能从本地持久化恢复会话列表与历史消息。
- **FR-010**: 网络异常或模型调用失败时，系统 MUST 不卡死，并提供可理解的错误提示与就地重试入口：重试 MUST 保留原用户消息，并在失败的那条助手消息上重新生成（不得为此追加新的用户消息副本）。
- **FR-011**: 系统 MUST 防止重复提交导致的重复生成，并避免消息乱序。首期 MUST 保证全局同一时刻至多一路生成；在一路未结束（含后台继续）时，其他会话发起的新生成 MUST 被拒绝，并给出可理解提示。
- **FR-012**: 系统 MUST 防止切换会话后出现串流污染（他会话的增量写入当前会话）。切换离开正在生成的会话时，该会话的生成 MUST 在后台继续，且增量 MUST 仅写入原会话；用户切回后 MUST 能看到进度或最终结果。
- **FR-013**: 模型调用 MUST 经由服务端发起；模型 API Key MUST 仅通过环境变量配置，不得写入代码库或暴露给浏览器。
- **FR-014**: 项目 MUST 提供环境变量示例文件（如 `.env.example`），说明所需配置项且不含真实密钥。
- **FR-015**: 系统 MUST 提供可显式开启的 Mock 模式。当 Mock 已显式启用时，MUST 走 Mock（即使已配置真实密钥）。当 Mock 未启用且已配置密钥时，MUST 走真实模型调用。当 Mock 未启用且无密钥时，MUST 给出可理解的配置错误提示，不得静默卡死。
- **FR-016**: 项目 MUST 至少为「消息状态」「流式解析」「会话管理」中的一类核心逻辑提供自动化测试。
- **FR-017**: 项目 MUST 提供完整使用说明文档，至少包含：启动方式、架构说明、已知限制。
- **FR-018**: 提交物 MUST 包含简短的 AI 工作记录（如 `AI_WORKLOG.md`），说明使用的 AI 编程工具、AI 完成的工作，以及发现并修正的错误；不得提交任何密钥或完整无关的 AI 对话记录。

### Key Entities

- **会话（Conversation）**：一次独立对话的容器；包含标识、标题或默认命名、创建/更新时间，以及有序消息列表。
- **消息（Message）**：会话内一条用户或助手发言；包含角色、内容、状态（等待/生成中/完成/失败/已取消等），以及可选的错误信息。失败后的重试复用同一条助手消息记录，不新建用户消息。已取消的助手消息保留停止前已生成的内容片段。
- **生成任务（Generation）**：针对某条助手消息的一次进行中请求；可被停止或失败；与所属会话绑定，用于并发与切换隔离。首期全局互斥（同一时刻最多一个活动生成任务）。
- **运行配置（Runtime Config）**：是否显式启用 Mock、是否配置模型密钥等。裁决顺序：显式 Mock → 真实密钥调用 → 否则配置错误提示。密钥永不暴露给前端。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 验收人员在 5 分钟内可按文档完成启动，并完成「创建会话 → 发送问题 → 看到逐步生成的回复」主路径。
- **SC-002**: 在生成过程中点击停止后，1 秒内可见生成终止且输入区可再次发送。
- **SC-003**: 在至少 2 个会话间切换往返后，各会话历史消息 100% 归属正确、无串会话内容。
- **SC-004**: 刷新页面后，刷新前已完成的会话与消息仍可被用户找到并继续对话。
- **SC-005**: 在模型失败场景下，用户总能看到失败说明与重试入口，且页面保持可操作（无整页卡死）。
- **SC-006**: 对前端构建产物进行文本检索，不出现模型 API Key 明文。
- **SC-007**: 在显式启用 Mock 的条件下（可不配置真实密钥），仍可完成本规格 P1 用户故事的验收场景。
- **SC-008**: 至少存在一类针对消息状态、流式解析或会话管理的自动化测试，且在标准测试命令下可重复通过。

## Assumptions

- 目标用户为单机本地使用的体验/验收人员；首期不要求账号登录、多用户隔离或云端同步。
- 持久化最低要求为浏览器本地存储即可；不要求服务端长期保存会话。
- 「OpenAI 兼容」类模型服务通过环境配置接入即可；首期只需对接一类兼容端点，不强制多供应商切换 UI。
- Mock 须由配置显式开启；不因「没有密钥」而自动进入 Mock（无密钥且未开 Mock 时应报错提示）。
- README 中的加分项（Regenerate/编辑重发/分叉、指标展示、状态机实现方式、多 Provider、E2E、Tool Call、移动端与无障碍深化等）**不作为首期必做范围**；仅在不损害核心正确性的前提下可择优实现。
- 技术栈可自选，但实现须前后端职责清晰，且满足密钥不落前端的约束。
- 题目建议投入约 75 分钟；规格以最低验收场景与核心要求为必达，不以页面视觉复杂度为目标。
- Constitution 文件仍为占位模板，本规格不依赖其中的具体治理条款；若后续正式化宪章，以实现期原则补充为准。

## Out of Scope（首期不做）

- 用户注册/登录与权限体系
- 会话云同步与多设备协同
- 文件/图片等多模态输入输出（文本对话为主）
- 必须以加分项全部做完才算交付
- 将真实 API Key 提交到仓库或文档
- **多会话并行生成**（升级版预留）：允许多个会话同时各自生成且互不串写；首期不做，待全局单路生成与串流隔离稳定后再升级
